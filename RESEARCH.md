# Reasearch - Coffee Shop API

## a. localhost vs Production
When I run my web app on localhost, it is only running on my own computer, usually through an address like http://localhost:3000. Other people cannot access it because localhost refers only to my machine, and my computer is not set up to serve that app publicly to the internet. A cloud platform like Render hosts the app on its own servers, which have public internet access, a real domain or URL, and networking configured so outside users can reach it. Render also keeps the application running for public requests instead of only while I am testing it locally. In short, localhost is for private development on my computer, while production on Render makes the app available to everyone online.

## b. Environment Variables
Environment variables are used instead of hardcoding values like port numbers, database connection strings, and API keys directly in the code because they keep sensitive information separate from the application code. By storing these values in a .env file, developers can easily change settings without modifying the code. This also helps prevent mistakes since the values are kept in one place instead of scattered throughout the project, reducing the risk of throwing errors. If a developer accidentally pushes a connection string with a password to a public GitHub repository, anyone on the internet is able to see their password. This could allow someone to access the database, steal/modify data, or use the credentials for god knows what. .env helps prevent this by keeping sensitive information out of the code that gets shared publicly by using it with .gitignore

## c. CORS
CORS is a browser security rule that controls when one website can request data from another domain. Browsers block these requests by default because allowing any website to access another site’s data could create security risks. The reason a browser blocks a request from one domain to another by default is for security reasons, as a malicious website could access the sensitive information from a user without them even realizing it. An example of configuring CORS would be if your website tries to request weather data from a weather API on a different domain, the browser will block it by default. The weather API would need to enable CORS to allow your website to access its data.

## d. Health Check Endpoints
A health check endpoint is a simple route in your app that lets a server or hosting platform check if your application is running correctly. It usually returns a basic response like “OK” to show the app is up and working. Hosting platforms like Render use health checks to make sure your app hasn’t crashed or stopped responding. If the platform checks the endpoint and doesn’t get a response, it knows something is wrong. If my deployed app on Render didn’t have a health check endpoint, Render might think the app is not working properly. This could cause the service to restart the app or mark the deployment as unhealthy.

## e. Free Tier Cold Starts
A cold start is when a hosted app has been inactive for a while and the service shuts it down to save resources. When someone tries to access the app again, the server has to start the app back up. Paid tiers handle this differently by keeping the server up at all times. In real production cold starts would be absolutely unacceptable, because the purpose of many apps are to be fast and effecient. If you need to wait to startup it can cause frustration and come off as very unprofessional.
